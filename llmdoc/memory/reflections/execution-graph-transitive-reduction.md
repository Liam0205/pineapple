# 执行图传递性归约下沉到构建阶段复盘

## Task
- 将 DAG 执行图的传递性归约从可视化渲染层下沉到 `dag.go` 的 `Build()` 阶段，使内部执行图和可视化都共享同一份最小边集。

## Expected vs Actual
- Expected outcome.
  - 图构建完成后即完成传递性归约，调度器与可视化都直接使用保持可达性不变的最小边集。
  - 相关测试主要验证 happens-before 语义与可达性，不依赖冗余直接边是否存在。
- Actual outcome.
  - 归约逻辑成功从 `visualize.go` 迁移到 `dag.go`，`Build()` 在拓扑排序前执行 `reduce(g)`，可视化层改为直接遍历 `Node.Succs`。
  - 新增了传递闭包等价、钻石图、屏障管线、线性链 noop 等归约测试，证明语义保持不变。
  - 两个既有测试因断言直接边而失败，后续改成验证可达性/语义后才与新所有权边界一致。

## What Went Wrong
- 上一轮复盘把“内部图保留完整边集以确保调度正确性”记录成了结论，但没有明确其前提是“当前尚未证明调度仅依赖可达性而非冗余直接边”。
- 既有测试中有部分断言落在实现细节上，例如 `hasPred` 这类直接边检查；图被优化后，这些测试虽然失败，但失败原因不是语义回归，而是断言层次过低。
- `visualize.go` 中的 `TestTransitiveReduction` 继续验证“内部完整图含冗余边”的前提，在归约所有权迁移后变成死断言，暴露出测试位置和行为所有权不一致。

## Root Cause
- 之前把一项阶段性设计选择写成了较稳定的架构判断，没有在复盘里标出适用条件与可推翻条件，导致后续优化时需要先推翻旧结论。
- 测试设计混用了“语义约束”和“结构实现”两个层次；当实现从完整边集切换为最小边集时，直接边断言自然变脆弱。
- 归约逻辑最初属于渲染层，因此相关测试也放在 `visualize.go`；迁移到构建层后，测试归属没有自动随所有权一起调整，需要显式重构。

## Missing Docs or Signals
- memory 中需要明确区分“当前设计选择”与“已验证的架构不变量”，尤其是像“完整边集是否为调度必需”这类可被后续证明推翻的判断。
- 稳定文档中应强调：执行语义依赖可达性/happens-before，而不是某条冗余直接边必须存在；这样新增测试时更容易选对断言层次。
- 需要更明确的测试归属信号：当某个算法从渲染层迁移到构建层时，其核心正确性测试也应一起迁移到对应所有者模块。

## Promotion Candidates
- 可推广到 `guides/` 或测试相关稳定文档：涉及 DAG 依赖关系时，优先断言可达性、拓扑顺序和语义约束，避免断言可被优化改变的直接边集合。
- 可推广到 `architecture/dag-engine.md`：构建后的执行图会进行传递性归约，内部图与可视化共享同一份最小边集；调度语义依赖可达性不变性。
- 仅保留在 memory：上一轮“内部图必须保留完整边集”的结论已被本次任务推翻，应作为认知修正历史保留，而不是再写入稳定文档。
